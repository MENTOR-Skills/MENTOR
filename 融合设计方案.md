# 本科生科研 Skills 组 · 融合设计方案

> 版本：v1.0 · 2026-07-14  
> 文档性质：正式设计报告  
> 依据文档：  
> 1.《本科生科研 Skills 组：当前功能规划共识》  
> 2.《同学科研 Skills 组 · 详细设计指南》  
> 定位：面向本科生的科研入门教练；不以代替学生完成科研或自动生成论文为目标。

---

## 0. 设计原则（全文有效）

1. **先有明确交付物，再决定调用哪些技能。** 下一轮工作以磁盘上的文件为准，不以聊天记忆为准。  
2. **关键步骤必须先经过用户确认，再写正式文稿。** 例如：课题范围未确认不得检索；付费论文未补全不得假装已精读。  
3. **机械检查优先。** 引用是否存在、中文文件编码是否正确、必填字段是否齐全，尽量用自动程序判断；模型负责理解、归纳与起草。  
4. **一项用户功能不等于一个技能。** 只有任务稳定、步骤可重复、需要读写文件或调用程序、有明确输入输出、有可检查的完成标准时，才单独做成技能。  
5. **基本版本优先落实「共识」中的五项用户功能。** 详细设计指南中的工程约定（工作区目录、确认关卡、编码检查、宿主适配）全部保留并吸收。

## 1. 实现的主要功能及技能分配

### 1.1 产品目标

本技能组帮助本科生完成四类真实需求：进入陌生研究方向、理解专题文献、整理本人已有科研材料、把可靠材料写成可用的科研表达。系统不承担「整夜自动实验并投稿」一类目标。

### 1.2 五项用户功能总览

| 编号 | 用户功能 | 一句话目标 | 主要交付物 |
|------|----------|------------|------------|
| F1 | 陌生领域入门路径 | 按学生背景给出可执行的学习顺序与第一个动手任务 | 学习地图、知识缺口说明、术语表、起步材料、首次练习 |
| F2 | 专题文献全景与关系图 | 围绕具体主题整理主线、代表工作与关系，并给出精读建议 | 结构化文献库、文献全景说明、精读清单、关系图网页 |
| F3 | 单篇论文深度阅读 | 帮助学生读懂问题、方法、证据与局限，必要时提取复现条件 | 阅读报告、论文卡片；复现模式另附检查清单 |
| F4 | 本地科研阶段总结 | 在授权范围内审计代码、日志与结果，形成可核对的阶段报告 | 材料索引、阶段报告、结果摘要、阻塞清单 |
| F5 | 基于研究材料的学术写作 | 把前述结构化证据写成特定用途文本 | 主张—证据对照表，以及阶段报告、组会材料等成稿 |

五项功能可以组合使用，**不要求每次从头走完**。例如：可直接深度阅读一篇论文；也可直接基于本地仓库做阶段总结。

### 1.3 功能到技能的分配

说明：下表中的「用户工作流技能」直接承接用户任务；「内部能力技能」可被多个工作流复用；「自动程序」负责易出错的机械步骤；「共用规程」全程生效，不单独生成科研正文。

#### F1 · 陌生领域入门路径

| 角色 | 名称 | 职责 |
|------|------|------|
| 用户工作流技能 | `domain-onboarding`（领域入门） | 询问背景与目标；生成个性化学习顺序；控制「先术语、后全景、再动手」的推进节奏 |
| 可选内容包 | `domain-packs/<领域编号>/` | 为常见领域提供预制术语、阅读顺序与小例子；**没有内容包时仍可根据对话生成入门材料** |
| 共用规程 | `academic-honesty`（学术诚信） | 禁止伪造「必读经典」名单；不确定处须标明 |

#### F2 · 专题文献全景与关系图

| 角色 | 名称 | 职责 |
|------|------|------|
| 用户工作流技能 | 由调度模块串联下列技能，不另设空壳「全景技能」 | 组织检索 → 筛选 → 精读入库 → 写全景说明 → 画关系图 |
| 执行技能 | `literature-search-download`（文献检索与下载） | 分级检索、合并去重、提出精读候选、仅对候选尝试开放获取下载 |
| 执行技能 | `literature-reader`（文献精读入库） | 把已读论文写入结构化文献库 |
| 执行技能 | `literature-landscape-writer`（文献全景撰写） | 基于文献库写出主线、分类与关系说明；**关系若由模型推断须显式标注，不得写成既成事实** |
| 执行技能 | `survey-visualizer`（文献关系可视化） | 只根据文献库生成网页图，不得编造图中数据 |
| 自动程序 / 裁判 | `citation-verifier`（引用核对） | 核对引用是否真实存在、关键字段是否完整 |
| 说明 | 完整长篇综述文章可作为后续扩展；**基本版本以「全景说明 + 精读清单 + 关系图」为完成标准**，不强制产出长综述 |

#### F3 · 单篇论文深度阅读

| 角色 | 名称 | 职责 |
|------|------|------|
| 用户工作流技能 | `paper-deep-read`（单篇深读） | 支持三种深度：快速判断是否值得读、深入分析、为复现提取条件 |
| 复用 | `literature-reader` | 深读结果必须写回结构化文献库，避免笔记孤立 |
| 自动程序 | PDF 解析辅助脚本（可放在深读或检索技能的 `scripts/` 下） | 提取文本、检查文件是否可读 |

#### F4 · 本地科研阶段总结

| 角色 | 名称 | 职责 |
|------|------|------|
| 用户工作流技能 | `progress-digest`（阶段进展整理） | 确认扫描范围与时间窗；建立材料索引；写阶段报告 |
| 自动程序（可选） | `list_recent_files.py` 等 | 列出近期改动文件，降低漏扫与乱扫风险 |
| 硬规则 | 写入技能正文 | 未获授权不得扫描整个用户主目录；不得因「有代码」就断定「实验已成功」 |

#### F5 · 基于研究材料的学术写作

> **扩大版写作中枢（审批结论已录入）：** F5 为全面写作中枢；**唯一写作技能** `academic-writing`（多文体包）+ 可选 `adversarial-lite`；不单独落地 `meeting-brief`。交付文件名 ASCII；Related Work 禁止 F5 从零写。详见 [docs/F5设计提纲-扩大版.md](./docs/F5设计提纲-扩大版.md)。**明确授权落地前不建技能目录。**

| 角色 | 名称 | 职责 |
|------|------|------|
| 用户工作流技能 | `academic-writing`（学术写作） | **唯一**写作工作流：多文体包（组会/PPT/阶段说明/技术文档/论文节改写/润色等）；强制主张—证据 |
| （已取消独立技能） | ~~`meeting-brief`~~ | 组会话术并入 `academic-writing` 文体包 |
| 内部能力技能 | 主张—证据整理（写作技能内**强制步骤**） | 成稿前必须先有 `_work/claims-evidence.md` |
| 内部能力技能 | `adversarial-lite`（挑刺审阅） | 可选；标准档不默认开；只提质疑，不擅自改终稿 |

### 1.4 横切能力（不属于某一项用户功能，但必须实现）

| 名称 | 类型 | 是否必要 | 说明 |
|------|------|----------|------|
| `campus-research-orchestrator`（调度模块） | 用户入口技能 | **必要** | 识别用户要进哪项功能、检查输入是否齐全、写入任务范围、调用对应技能、在需确认处停下 |
| `research-workspace`（工作区约定） | 规程 + 可选初始化脚本 | **必要** | 统一目录与状态字段，保证跨次会话可续做 |
| `academic-honesty`（学术诚信） | 共用规程（Markdown） | **必要** | 全程禁止伪造引用、伪造实验结果、伪造已读文献 |
| 系统级总说明（见第 2 章） | Markdown 索引，不是独立「聊天系统提示产品」 | **建议具备** | 给智能体冷启动时的总路由与禁令索引 |
| 独立「监管模块」技能 | — | **基本版本不单独立项** | 监管职责拆到：诚信规程 + 引用核对程序 + 编码检查程序 + 各技能内的确认关卡；避免再做一个空泛的「总监管」技能 |

### 1.5 明确不单独实现的能力

1. **独立的「科研任务澄清」技能**：只作为调度入口的必要提问与范围写入，不单独生成长篇任务分析。  
2. **独立的「下一步行动建议」技能**：下一步建议写在各功能流程结尾，必须附着于当次已有证据，避免空泛建议。  
3. **基本版本不单独实现「从研究缺口自动出题」**：若需要，作为 F2/F3 结束后的简短出口步骤即可。

## 2. 组织层次设计

### 2.1 层次总览

本仓库分为三层，职责不可混用：

```text
第一层  技能包（版本管理，由作者维护）
        └── 安装到 Cursor / Claude Code 等环境后被调用

第二层  领域内容包（可选，由队伍维护）
        └── 为常见方向提供本地细化材料

第三层  用户课题工作区（每次任务生成，由智能体与用户共同写入）
        └── 存放范围说明、文献库、报告等；技能包升级不得覆盖此目录
```

### 2.2 技能包目录与文件职责

```text
campus-research-skills/
├── README.md                          # 给人看的安装与快速开始
├── AGENTS.md                          # 给智能体的总索引（轻量系统说明）
├── SKILLS_CATALOG.md                  # 技能一览（给人读）
├── shared/                            # 全组共用说明（Markdown）
│   ├── honesty-checklist.md           # 成稿前诚信勾选清单
│   ├── effort-contract.md             # 深度档：轻量 / 标准 / 深入 的行为上限
│   ├── encoding-utf8.md               # Windows 中文防乱码写法与检查命令
│   ├── workspace-layout.md            # 工作区目录说明
│   └── platform-notes.md              # 不同软件宿主的安装与调用差异
├── skills/                            # 可安装技能
│   ├── campus-research-orchestrator/
│   ├── research-workspace/
│   ├── academic-honesty/
│   ├── domain-onboarding/
│   ├── literature-search-download/
│   ├── literature-reader/
│   ├── paper-deep-read/
│   ├── literature-landscape-writer/
│   ├── citation-verifier/
│   ├── survey-visualizer/
│   ├── progress-digest/
│   ├── academic-writing/
│   ├── meeting-brief/
│   └── adversarial-lite/
├── domain-packs/                      # 可选领域内容包
│   └── _template/
└── examples/                          # 演示用示例话题（可选）
```

单个技能的标准结构：

```text
skills/<技能名>/
├── SKILL.md                 # 必选：何时用、步骤、确认关卡、交付约定、禁止项
├── references/              # 可选：长模板、字段说明（需要时再读）
└── scripts/                 # 可选：Python 等自动程序
```

**不要放入技能包的内容：** 大批量 PDF、虚拟环境、模型权重、用户隐私数据。

### 2.3 每个文件「用什么类型、干什么」

| 路径 | 文件类型 | 用途 |
|------|----------|------|
| `README.md` | Markdown | 安装、非目标声明、三条最小演示话术 |
| `AGENTS.md` | Markdown | 智能体冷启动：功能路由表、交付文件名总表、停下等待用户时的恢复说明 |
| `shared/*.md` | Markdown | 诚信、深度档、编码、目录、宿主差异；被各技能引用，避免多处漂移 |
| 各 `SKILL.md` | Markdown（含简短元信息） | **运行时行为的唯一正文来源**：步骤可勾选、确认关卡、输入输出路径 |
| `references/*.md` | Markdown | 写作模板、字段词典、挑刺启发式等长材料 |
| `scripts/*.py` | Python 程序 | 检索、核对引用、生成关系图、列出近期文件、检查 UTF-8 |
| `domain-packs/**` | YAML + Markdown + 可选静态网页/小例子 | 本地细化信息，不是通用推理逻辑 |
| 用户工作区中的 `*.md` | Markdown | 给人阅读的报告与说明 |
| 用户工作区中的 `references.json` 等 | JSON | 给后续技能读取的结构化数据 |
| 用户工作区中的 `viz/*.html` | HTML | 仅供浏览的关系图，数据来自 JSON |

### 2.4 系统提示、调度模块、监管模块：要不要单独做

#### （1）系统提示 / 总说明

| 判断 | 结论 |
|------|------|
| 是否必要 | **有必要，但保持轻量** |
| 推荐形态 | 仓库根目录 `AGENTS.md` + `shared/` 共用说明；**不**另维护一份与技能正文互相矛盾的「超级系统提示」 |
| 写什么 | 本产品是什么、不是什么；如何根据用户意图进入 F1–F5；必须遵守的全局禁令；指向各 `SKILL.md` |
| 不写什么 | 具体检索策略、具体综述结构（这些属于对应技能） |

理由：智能体需要稳定的冷启动索引；但若把全部业务步骤塞进系统提示，会造成与技能正文双源维护、极易漂移。

#### （2）调度模块

| 判断 | 结论 |
|------|------|
| 是否必要 | **必要** |
| 推荐形态 | 独立技能 `campus-research-orchestrator`，正文为 Markdown 规程；可用少量脚本做编码检查 |
| 职责边界 | 识别功能、检查输入、维护工作区状态、调用子技能、在确认点停下；**不负责生成科研正文** |
| 与「共识」关系 | 对应共识中的「轻量总控」 |

#### （3）监管模块

| 判断 | 结论 |
|------|------|
| 是否做成单独大技能 | **基本版本不建议** |
| 监管如何落地 | 拆成三件实事：① `academic-honesty` 规程；② `citation-verifier` 等自动核对；③ 各技能内的确认关卡与「主张—证据」强制步骤 |
| 可选增强 | 若比赛演示需要「显式审阅角色」，使用已有的 `adversarial-lite`（挑刺），仍不上升为笼统监管中枢 |

理由：单独的「监管模块」若没有稳定输入、工具动作与交付物，容易退化成一段空泛提示；与共识中的技能判定标准不符。

### 2.5 用户课题工作区（运行时产物）

默认路径：`campus-research-output/<课题简称>/`（允许用户指定根路径）。

```text
campus-research-output/<课题简称>/
├── scope.md                 # 主题、时间范围、深度档、用户确认事项、禁止事项
├── findings.md              # 当前认知摘要（可跨周更新）
├── research-log.md          # 追加式工作记录（做了什么、为何停下）
├── _work/                   # 过程文件（可重建）
│   ├── WAITING_*.md         # 正在等待用户完成的事项说明
│   ├── shortlist.json       # 精读候选
│   ├── claims-evidence.md   # 主张与证据对照
│   └── state.yaml           # 可选：当前功能、确认关卡状态
├── pdfs/
│   ├── auto/                # 自动下载的开放获取全文
│   └── user/                # 用户自行补充的全文
├── references.json          # 结构化文献库（后续技能以此为准）
├── reading-reports/         # 单篇阅读报告（给人读）
├── learning-map.md 等       # F1 交付
├── literature-landscape.md 等  # F2 交付
├── progress-report.md 等    # F4 交付
├── draft.md / meeting-*.md  # F5 交付
└── viz/                     # 关系图网页
```

**约定：** 结构化数据以 JSON 为准；Markdown 报告是给人阅读的说明文。单篇深读必须回写 `references.json`，避免笔记无法进入专题全景。

### 2.6 工作深度档（轻量 / 标准 / 深入）

| 档位 | 适用场景 | 行为约束 |
|------|----------|----------|
| 轻量 | 今晚组会、快速判断一篇论文 | 少检索、短文稿；必须标明局限 |
| 标准 | 默认 | 完整确认关卡；精读候选需可核对；主张挂证据 |
| 深入 | 开题周、正式相关工作整理 | 更广检索、更长精读、启用挑刺、关系图必出 |

未指定时默认「标准」。用户说「随便写写」时，应建议降为「轻量」并请用户确认。

## 3. 工作流设计

### 3.1 总流程

```text
用户提出需求
    → 调度模块：识别属于 F1–F5 哪一项（含混则只问一个澄清问题）
    → 检查或创建工作区；确认 scope.md
    → 加载学术诚信规程
    → 调用对应技能链
    → 在确认关卡写入 WAITING 说明并停下（如需用户补 PDF、确认主张）
    → 保存结构化产物与给人阅读的报告
    → 在流程结尾给出「附着于本次证据」的下一步建议
```

两条常见使用路径（可跳步）：

```text
知识与文献路径：
  入门路径（F1）→ 专题文献全景（F2）→ 单篇深读（F3）→ 学术写作（F5）

真实项目路径：
  本地代码 / 日志 / 实验结果 → 阶段总结（F4）→ 学术写作或组会材料（F5）
```

### 3.2 各功能流程要点

#### F1 入门路径

1. 询问学生已有课程背景、目标实验室或时间预算。  
2. 若存在匹配的领域内容包则加载；否则基于对话生成。  
3. 产出学习地图、知识缺口、术语表、起步材料、首次练习。  
4. 推进节奏：未掌握核心术语前，不推荐「全面综述」。  
5. 结尾建议：带着一个自拟小问题进入 F2。

#### F2 文献全景

1. 确认主题、年份范围、深度档；不代填用户未指定的会议或期刊名单。  
2. 分级检索 → 合并去重 → 提出精读候选（深入档必须请用户确认候选）。  
3. 仅对候选尝试开放获取下载；遇付费墙则停下，请用户放入 `pdfs/user/`。  
4. 精读结果写入 `references.json`。  
5. 撰写文献全景说明与精读清单；生成关系图网页。  
6. 运行引用核对；失败则回改文稿或标记未核实，不得静默删除引用。

#### F3 单篇深读

1. 用户给出论文 PDF、链接或已在文献库中的编号。  
2. 按选定深度产出阅读报告与论文卡片。  
3. 复现深度另附数据、参数与环境检查清单。  
4. 回写文献库，供 F2/F5 继续使用。

#### F4 阶段总结

1. 确认扫描根路径与时间窗（如「自上次组会以来」）。  
2. 建立材料索引：路径、一句话用途、是否已读。  
3. 综合为阶段报告：目标、已完成（每条挂证据）、结果摘要、阻塞、后续动作。  
4. 忽略规则：依赖目录、版本库内部、模型权重、数据集本体、密钥与环境变量文件。

#### F5 学术写作

1. 选择文体：阶段技术说明、组会一页纸、口播要点、预期问答等。  
2. 强制先写主张—证据对照表。  
3. 标准档与深入档对高风险主张请用户确认。  
4. 成稿后可选挑刺审阅；作者修订后更新工作记录。

### 3.3 场景叙述

**场景 A：一周后进组，方向完全陌生**  
学生说明本科课程背景与目标方向。调度进入 F1，生成个性化入门材料与第一个动手练习。学生完成术语关后，带着自拟小问题进入 F2，得到专题全景与三到五篇精读建议，再对其中一篇跑 F3。

**场景 B：两天后组会，相关工作还没理清**  
调度进入 F2（标准档）。确认范围后检索并筛选；若有付费论文则停下等学生补文件；随后产出全景说明、精读清单与关系图。再进入 F5 的组会文体，压缩为一页纸与预期问答；问答中的每条关键判断指向文献库或本地材料。

**场景 C：这周实验很乱，需要阶段总结并汇报**  
学生指定 `runs` 目录与主要训练脚本。调度进入 F4，生成材料索引与带证据链接的阶段报告。再进入 F5/组会材料：一页纸列出卡点与需要导师拍板的问题（建议不超过三个）。对「缺少消融实验」一类弱点，可启用挑刺技能预演导师可能的追问。

**场景 D：只要读懂这一篇，并判断能不能复现**  
调度直接进入 F3 的复现深度。产出阅读报告、论文卡片与复现检查清单；同时写入文献库，供以后做专题全景时复用。

**场景 E：已有阶段报告，要写成可提交的技术说明**  
调度进入 F5。读取 F4 产物与可选的文献库；先完成主张—证据表，再起草技术说明；对无证据支撑的句子标出待补项，而不是编造结果。

## 4. 其他设计注意事项（含兼容性）

### 4.1 学术与安全底线

1. 不得伪造引用、页码、实验数值、下载状态。  
2. 模型推断的论文关系必须标明「推断」，不得写成数据库事实。  
3. 不得绕过出版社付费限制获取全文。  
4. 不得在未授权时扫描用户主目录或敏感路径。  
5. 忽略并不得外传密钥、令牌、含隐私的配置。

### 4.2 中文与 Windows 环境

1. 中文交付文件必须使用 UTF-8 保存。  
2. 在 Windows 上，正式交付前运行编码检查程序。  
3. 脚本路径使用相对路径，禁止写死个人电脑绝对路径。

### 4.3 兼容性策略：一份正文，薄适配层

| 层 | 策略 |
|----|------|
| 规程与交付约定 | 全宿主共用（技能正文、模板、脚本逻辑） |
| 安装与调用方式 | 写入 `shared/platform-notes.md`，技能正文用中性表述 |
| 工具能力下限 | 读文件、写文件、运行终端 Python、访问网络；不把某一家专用插件当默认依赖 |
| 可选能力 | LaTeX 编译、跨模型互审标为可选 |

**Cursor：** 技能安装到个人或项目的 skills 目录；依靠技能描述中的触发词被发现；用 Python 运行脚本。  

**Claude Code：** 可用项目内 skills 目录或符号链接指向本仓库；常见以斜杠命令调用；可在项目说明中写一句「科研任务优先遵循本仓库 AGENTS.md」。  

**清小搭或校内智能共创平台（若上架）：** 用工作流节点对应调度步骤；用选择型/输入型交互对应确认关卡；领域内容包装入平台文档库；禁止把 Cursor 专用路径写死进平台节点。

### 4.4 依赖与降级

| 依赖 | 基本要求 | 达不到时的降级 |
|------|----------|----------------|
| Python 3 | 文献检索、核对、可视化需要 | 仅开放「纯模板写作 / 入门问答」子集，并明确声明 |
| 网络（开放文献接口） | F2 需要 | 允许用户只提供本地 PDF 进入深读 |
| LaTeX | 非必须 | 默认 Markdown 交付 |
| 第二模型 | 非必须 | 挑刺技能在同一模型会话内完成 |

### 4.5 交付与命名冻结

基本版本对外交付文件名建议冻结如下（实现前勿随意双轨命名）：

| 功能 | 建议文件名 |

## 5. 子模块划分与设计计划

### 5.1 子模块一览

| 模块 | 包含内容 | 主要产出类型 |
|------|----------|--------------|
| M0 基础底座 | 调度、工作区、诚信、共用说明、编码检查 | Markdown 规程 + 少量 Python |
| M1 文献理解 | 检索下载、精读入库、单篇深读、全景撰写、引用核对、关系图 | Markdown + Python + JSON/HTML |
| M2 本地进展 | 阶段进展整理 | Markdown + 可选 Python |
| M3 科研表达 | 学术写作、组会材料、挑刺审阅 | Markdown 模板为主 |
| M4 入门教练 | 领域入门技能 + 至多一个示范领域内容包 | Markdown + 可选 YAML/小例子 |

### 5.2 分批实现计划

| 批次 | 目标 | 模块 | 完成标准（可演示） |
|------|------|------|--------------------|
| 第一批 | 搭好底座，能路由、能停、能续跑 | M0 | 用户一句话能进入正确功能；`scope.md` 与工作记录可写；确认关卡可挂起 |
| 第二批 | 单篇深读 + 文献库回写 | M1 子集 | 给定 PDF 产出阅读报告，且 `references.json` 有对应条目 |
| 第三批 | 本地阶段总结 + 组会一页纸 | M2 + M3 子集 | 指定目录产出带证据路径的阶段报告与组会一页纸 |
| 第四批 | 专题文献全景（检索到关系图） | M1 其余 | 精读清单、全景说明、关系图、引用核对可跑通；付费墙可正确停下 |
| 第五批 | 个性化入门路径 | M4 | 按背景生成五类入门文件；可选加载一个示范内容包 |
| 第六批 | 写作补全与挑刺 | M3 其余 | 主张—证据表强制出现；阶段写作与预期问答可用；挑刺只输出批评稿 |

说明：批次顺序兼顾「共识」的产品叙事与现有文献脚本资产。第二、三批演示成本低、贴近本科生进组场景；第四批复用已有检索与可视化能力；入门与完整写作能力随后补齐。

### 5.3 每批设计时必须写清的五项

对每一个即将实现的技能，设计文档或 `SKILL.md` 中至少写清：

1. 输入是什么（用户必须提供什么）；  
2. 有序步骤是什么；  
3. 哪些步骤必须停下来问人；  
4. 交付文件路径与字段；  
5. 怎样算完成、失败时如何退回上一步。

### 5.4 与现有演示技能的关系

若仓库中已有文献调研演示技能，第一至第四批应优先迁入并改造：将「长篇综述为唯一完成标准」调整为「全景说明 + 精读清单 + 关系图」；将精读能力同时暴露为 F3 的独立入口。

## 6. 扩展计划

**F5 扩大版写作中枢（提纲，优先于下列第 1 项中的「组会/PPT 写作」拆分）：** 产品意向已将 F5 从基本版薄写作层升级为全面写作中枢（组会稿、PPT、论文章节、润色、技术文档等）。详细文体表、技能拆分与分期见 [docs/F5设计提纲-扩大版.md](./docs/F5设计提纲-扩大版.md)；**审批前不建技能目录**。基本版收窄描述以该提纲为准待批。

### 6.1 基本版本之后、确有需求的扩展方向

下列方向均建立在五项功能已经可演示之后，按真实使用反馈择优，不提前铺开。

1. **完整相关工作 / 综述成稿**  
   在文献库与全景说明已稳定后，增加「压成论文相关工作章节」或「短综述文章」模式，并强制走主张—证据与引用核对。

2. **开题问题收窄**  
   根据文献库中的缺口、矛盾与未覆盖类别，生成三到五个可检验的小问题，由学生选定后写入任务范围。此能力在基本版本中只作为流程结尾建议，扩展期可升级为独立轻量技能。

3. **学术邮件与合作沟通文体**  
   在证据约束下起草请求基线代码、约讨论等邮件；禁止捏造已有结果。

4. **复现实验助手（仍非自动实验室）**  
   在 F3 复现清单基础上，增加「对照本地仓库缺什么」的差异检查；仍不默认整夜改代码刷指标。

5. **组内模板与课程作业适配**  
   支持实验室报告模板、课程开题表格等本地格式，通过内容包或模板目录扩展，而不是改调度逻辑。

6. **多宿主与平台上架打磨**  
   在 Cursor 与 Claude Code 实测通过后，再做清小搭节点映射与报告导出。

### 6.2 如何把「偏通用的技能」细化到具体研究方向

目标：通用技能负责「怎么做」；本地材料负责「这个方向带什么」。换方向时尽量只加内容，不改引擎。

#### （1）领域内容包（推荐主路径）

为每个方向建立内容包，例如 `domain-packs/embodied-ai/`，建议包含：

| 文件 | 作用 |
|------|------|
| `pack.yaml` | 编号、标题、适用读者、预计学时、维护者 |
| `curriculum.md` | 有序阅读路径（链接公开资料或本地样例，不塞全文） |
| `glossary.md` | 核心术语（建议控制数量，避免变成词典堆砌） |
| `checklist.md` | 「是否具备开题最小条件」自检 |
| `starter-questions.md` | 练手级问题示例（须标明是示例，不替学生选题） |
| `mini-example/` | 最小可运行例子或指向外部仓库的说明 |
| `viz/`（可选） | 概念关系静态图 |

`domain-onboarding` 读取内容包后，再结合学生背景生成个性化缺口与学习顺序。这样，**通用入门技能不必为每个实验室改代码**。

#### （2）课题级配置（每个学生/每个题目一份）

在 `scope.md` 中记录：主题、深度档、输出语言、用户指定的期刊会议偏好、允许扫描的本地路径、选用的领域内容包编号。技能通过**读配置改变行为**，避免为每个实验室复制一份技能。

#### （3）结构化文献库的持续积累

F2、F3 持续写入 `references.json`。后续全景、写作、组会都读取同一库。方向越做越深时，细化来自「本课题已读文献与本地实验证据」，而不是靠更长的通用提示词。

#### （4）实验室私有补充（可选）

在用户工作区或内容包中增加只读的「组内约定」文档，例如：常用基线、禁止踩的坑、报告章节顺序。调度模块只负责发现并遵守这些本地文件，不把组内秘密写进公开技能包。

#### （5）细化时的边界

1. 内容包维护者不得在包内写死付费全文。  
2. 示例问题必须标注为示例。  
3. 通用技能升级时，不得覆盖用户课题工作区与实验室私有补充材料。

| 工程 | 技能判定标准、轻量总控 | 目录、深度档、宿主兼容、分批计划 | 全部保留 |
