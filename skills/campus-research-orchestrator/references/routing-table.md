# F1–F5 功能匹配参考

> 被 `campus-research-orchestrator` 步骤 2 使用。
> 与 `AGENTS.md` §1 功能索引保持同步；本文件为详细扩展版（增加触发词和边界案例）。

---

## 匹配原则

**无硬编码优先级。** Agent 根据用户表述的真实意图判断需要哪个功能：

- 单一意图清晰 → 直接路由
- 明显涉及多个功能 → 建议串行顺序并请用户确认
- 含混（可合理归入两个功能）→ 问一个澄清问题
- 完全无法匹配 → 列出五项功能让用户选

---

## 各功能触发词与边界

### F1 · 陌生领域入门

| 维度 | 内容 |
|------|------|
| 触发词 | 入门、不了解、完全陌生、怎么开始、学什么、零基础、新手、new to、getting started、beginner、onboarding |
| 边界 | 用户对该领域缺乏系统认知，需要从基础概念开始 |
| 不应路由到 F1 | 用户只需找特定论文、只要总结已有实验、已对该领域有系统了解 |
| 入口技能 | `domain-onboarding` |

### F2 · 专题文献全景与关系图

| 维度 | 内容 |
|------|------|
| 触发词 | 找论文、文献、综述、相关工作、有哪些经典、关系、literature、survey、review、papers |
| 边界 | 用户需要围绕一个具体主题系统整理文献 |
| 不应路由到 F2 | 用户只要深读单篇已指定的论文（F3）、只要总结本地实验（F4） |
| 入口技能 | `literature-search-download` |

### F3 · 单篇论文深度阅读

| 维度 | 内容 |
|------|------|
| 触发词 | 读这篇、这篇论文、读懂、复现、paper、read、deep read、reproduce |
| 边界 | 用户给出具体论文 PDF/链接/编号，需要深入理解或提取复现条件 |
| 不应路由到 F3 | 用户还没有具体论文、需要先检索（F2） |
| 入口技能 | `paper-deep-read` |

### F4 · 本地科研阶段总结

| 维度 | 内容 |
|------|------|
| 触发词 | 总结、进展、这周干了什么、阶段、日志、实验结果、progress、summarize |
| 边界 | 用户有本地实验/代码/日志需要梳理成阶段报告 |
| 不应路由到 F4 | 没有本地材料、纯粹讨论文献、只要写组会材料但材料已齐（F5） |
| 入口技能 | `progress-digest` |

### F5 · 基于研究材料的学术写作

| 维度 | 内容 |
|------|------|
| 触发词 | 写、组会、汇报、PPT、一页纸、润色、技术说明、技术文档、write、meeting、polish |
| 边界 | 用户有可核对材料，需要写成特定用途文稿 |
| 不应路由到 F5 | 用户要从零写 Related Work / 长综述（→ F2 `survey-writer`） |
| 入口技能 | `academic-writing` |

## 多意图组合（常见串行模式）

| 用户表述 | 建议顺序 |
|---------|---------|
| 「先入门再找论文」 | F1 → F2 |
| 「读完论文写组会材料」 | F3 → F5 |
| 「总结这周实验，写组会一页纸」 | F4 → F5 |
| 「找论文、精读、写 Related Work」 | F2 → F3 → F5（Related Work 部分需先有 F2 `survey-writer` 产物） |

串行时每个功能独立停车、独立完成；上一个功能的产出作为下一个的输入。

## 维护约定

1. 本表与 `AGENTS.md` §1 功能索引保持同步；新增触发词时两处一起改
2. 不在此表中写「A 功能优先于 B 功能」——匹配由 Agent 语义理解完成
3. 边界案例持续积累（如用户说「总结论文」→ 是总结实验 F4 还是总结文献 F2？→ 问澄清问题）
