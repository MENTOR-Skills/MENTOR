# 分支约定

## 1. 长期分支

| 分支 | 用途 |
|------|------|
| `main` | 可给全员对齐的稳定内容：设计文档、骨架、已合并的技能。默认保护，不直接堆半成品 |

## 2. 功能分支命名

每人从最新 `main` 拉出自己的分支，命名建议：

```text
feat/<范围>-<简述>
```

本周推荐：

| 负责人范围 | 分支名示例 |
|------------|------------|
| F1 入门 | `feat/f1-onboarding` |
| F2 文献全景 | `feat/f2-landscape` |
| F3 单篇深读 | `feat/f3-deep-read` |
| F4 阶段总结 | `feat/f4-progress-digest` |
| 横切 + 学术写作 | `feat/crosscut-writing` |
| 仅改文档 / 协议 | `docs/<简述>` |
| 修复 | `fix/<简述>` |

说明：`<范围>` 用小写英文或功能编号即可，避免中文分支名（部分工具兼容性差）。

## 3. 日常流程

1. 开工前：`git checkout main` → `git pull` → `git checkout -b feat/...`
2. 小步提交，提交说明写清「改了哪个技能 / 哪类文档」
3. 合并前向 `main` 开 Pull Request（或组织内约定的合并方式）
4. PR 描述中列出：**新增/修改的文件路径**、**如何演示一句用户话术**、**产出了哪些约定文件名**
5. 合并后删除已合并的功能分支（可选）

## 4. 冲突与改协议

- 只改自己负责目录时，冲突应很少
- 若需改 `docs/接口与协议.md` 中的冻结项：先开 `docs/...` 短 PR，合并后再改功能代码
- 共用文件（如根目录 `README.md`、调度技能）改动时，在群里知会相关负责人

## 5. 暂不要求（可下周再加）

- 强制代码审查人数
- 分支保护规则与 CI

当前阶段以「能并行、能回滚、能看清谁改了什么」为准。
