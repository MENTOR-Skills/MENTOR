# F3 单篇论文深读开发总结

> 更新时间：2026-07-20  
> 功能：F3 单篇论文深读（`paper-deep-read`）  
> 当前状态：**可演示**

## 1. 这次开发解决了什么

F3 面向“读懂一篇指定论文”的场景。当前实现接收本地 PDF、可靠全文链接、论文标题或 `references.json` 中的论文 ID，产出面向人的阅读报告，并把结构化笔记安全写回同一份文献库。

本轮开发重点不是增加更多说明，而是收紧职责：

1. 将 `SKILL.md` 从 126 行压缩到 74 行，只保留触发范围、门控、流程、交付契约和命令入口。
2. 将 `references/report-template.md` 设为报告结构、语言风格和证据纪律的唯一权威来源。
3. 将样例目录收缩为三个最终交付，不再混入输入夹具、状态文件、日志和中间 JSON。
4. 补齐 `references.json` 的安全写回脚本及失败路径测试。
5. 修复 PDF 渲染资源路径、临时目录、危险输出和旧 PDF 保护问题。

## 2. 当前设计

### 2.1 技能与模板的职责边界

- `skills/paper-deep-read/SKILL.md`：回答何时调用、何时停车、按什么顺序执行、最终交什么。
- `skills/paper-deep-read/references/report-template.md`：定义报告写什么、如何解释、证据表怎么组织。
- `skills/paper-deep-read/assets/`：保存 PDF 排版使用的 LaTeX 模板和元数据。
- `skills/paper-deep-read/scripts/render_pdf.py`：只负责从 Markdown 安全派生 PDF。
- `skills/literature-reader/scripts/upsert_reference.py`：负责把单篇结构化结果合并回共享文献库。

不要把报告章节、完整 JSON 字段表、依赖安装教程或 PDF 实现细节重新复制回 `SKILL.md`。

### 2.2 阅读状态模型

当前全局深度档固定为 `standard`。`rough` 和 `deep` 是标准档内部的阅读阶段，不是轻量档或深入档：

```text
获得全文 → stage: rough → 用户确认 → stage: deep
全文不可得 → waiting_user_pdf
                   ├─ 用户补 PDF → fulltext
                   └─ 用户明确接受 → abstract_only
```

- `rough`：先形成整体地图，说明问题、核心点子、框架、最强证据和最大疑点。
- `deep`：复核粗读判断，再展开方法、公式、实验口径、证据强度、局限和复现信号。
- `abstract_only`：只能写摘要直接支持的内容，不得补公式、算法细节或摘要未给出的实验数字。

### 2.3 交付契约

F3 的冻结主交付是：

- `reading-reports/<id>.md`
- 回写后的工作区根 `references.json`

可选派生物：

- `reading-reports/<id>.pdf`
- 用户明确要求复现准备时的 `reproduction-checklist.md`

`<id>` 必须与 `references.json` 中该论文的 `id` 一致。Markdown 是主交付，PDF 渲染失败不能被当作论文阅读失败。

## 3. 关键工程实现

### 3.1 安全文献库写回

命令：

```bash
python skills/literature-reader/scripts/upsert_reference.py references.json _work/f3-paper.json
```

当前脚本保证：

- 以论文 `id` 为主键新增或更新；
- 递归合并 `reading_notes`，空新值不覆盖有效旧值；
- 保留人工扩展字段和已有 `relations`；
- 防止 `fulltext` 被降级为 `abstract_only`；
- DOI 或 arXiv ID 与其他论文冲突时拒绝写回；
- 文献库损坏或写入失败时保留原文件；
- 使用同目录临时文件和原子替换。

### 3.2 PDF 渲染

命令：

```bash
python skills/paper-deep-read/scripts/render_pdf.py reading-reports/<id>.md
```

渲染器当前会：

- 从 `assets/paper-note.tex` 和 `assets/metadata.yaml` 加载排版资源；
- 拒绝输入输出同路径以及非 `.pdf` 输出；
- 在目标目录的独占临时目录中构建；
- 渲染失败时保留 Markdown 和既有 PDF；
- 成功后原子替换目标 PDF并清理临时目录；
- 缺少 Pandoc 或 XeLaTeX 时给出明确错误，不破坏主交付。

## 4. BERT 样例试跑

样例目录：`examples/f3-sample-run/`

目录严格只保留：

```text
examples/f3-sample-run/
├── references.json
└── reading-reports/
    ├── 1810.04805.md
    └── 1810.04805.pdf
```

该样例使用 BERT（`1810.04805`），基于 ACL Anthology 正式论文和 Google Research 官方仓库核对。来源链接与复核记录已写入 `references.json` 和阅读报告，因此不再单独保留 `source.md`。

当前 PDF 为 A4、7 页、141271 字节，带可检索文字层；已逐页检查，没有发现表格越界、文字截断或页面重叠。

样例目录不要再加入 `input/`、`expected/`、`_work/`、`scope.md`、`research-log.md` 或测试夹具。自动化测试需要的临时输入应由 `tests/` 在临时目录中构造。

## 6. 当前边界

- 当前状态是“可演示”，不代表已覆盖所有论文类型或完成大规模真实用户盲测。
- F3 不运行训练、推理或复现实验；最多在用户授权后生成复现条件清单。
- 全文不可得时必须停车或经用户同意降为 `abstract_only`，不得依靠模型记忆补全文内容。
- PDF 是可选派生物；没有 Pandoc/XeLaTeX 时仍应交付有效 Markdown。
- 当前只有 BERT 一个正式成品样例；增加新样例前应先证明它覆盖了新的论文类型或失败路径。

## 7. 后续修改约定

后续开发 F3 时请遵守以下顺序：

1. 报告内容或写作风格变化：先改 `references/report-template.md`。
2. 流程、门控或交付契约变化：再改 `SKILL.md`。
3. 文献库字段合并行为变化：改 `upsert_reference.py`，同时补 `test_upsert_reference.py`。
4. PDF 行为变化：改 `render_pdf.py`，同时覆盖成功和失败路径。
5. 样例变化：仍只保留三个最终文件，测试输入放临时目录。
6. 任何“测试通过”声明必须附实际命令和当次运行结果，不能只写在概况文档中。

如果交付名或 `references.json` 最小字段发生变化，必须先同步 `docs/接口与协议.md`、`AGENTS.md` 和对应契约测试，避免多个技能读写口径漂移。
