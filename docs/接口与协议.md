## 1. 仓库内目录职责（技能包）

```text
MENTOR/
├── README.md
├── 融合设计方案.md
├── docs/                    # 协作与协议文档
├── skills/                  # 可安装技能（每个子目录一个技能）
├── shared/                  # 全组共用说明（诚信清单、目录说明等）
├── domain-packs/            # 可选：按研究方向细化的内容包
│   └── _template/           # 新建领域包时的模板目录
└── examples/                # 演示用样例（假课题、话术等）
    └── sample-topic/
```

| 路径 | 谁维护 | 说明 |
|------|--------|------|
| `skills/` | 各功能负责人 | 只放技能定义与脚本，不放用户论文全文 |
| `shared/` | 横切负责人牵头 | 多技能共用的 Markdown 说明，避免各写一份漂移 |
| `domain-packs/` | F1 牵头 | 通用入门技能读这里的内容；无包时仍可对话生成 |
| `examples/` | 任意人可补 | 联调样例；大 PDF 默认不进库（见 `.gitignore`） |
| `docs/` | 全员 | 分支、接口、分工等协作文档 |

**不要写入本仓库的内容：** 用户真实课题产出目录、密钥、虚拟环境、模型权重、大批量 PDF。

## 5. 编码与安全底线

1. 中文 Markdown / JSON 使用 UTF-8 保存。  
2. 不得伪造引用、实验数值、下载状态。  
3. 模型推断的论文关系须标明「推断」。  
4. 不得在未授权时扫描用户主目录；忽略密钥与 `.env`。  
5. 不得把付费全文或密钥提交进本仓库。

## 6. `references.json` 最小字段

文献库为 JSON 文件；建议顶层为对象，内含 `papers` 数组（或等价列表）。**每一篇论文条目**至少包含下表「必填」列。

| 字段 | 必填 | 含义 |
|------|------|------|
| `id` | 是 | 稳定编号（优先用开放获取库 id、arXiv id；否则团队内手编且全程不变） |
| `title` | 是 | 标题 |
| `authors` | 是 | 作者列表（字符串数组，或统一约定的作者对象数组；同一仓库内保持一种写法） |
| `year` | 是 | 发表年（未知可写 `null`，但字段要有） |
| `venue` | 是 | 出版情况：期刊 / 会议 / 预印本平台等名称；未知写空字符串并在笔记中说明 |
| `url` | 建议 | 可访问链接 |
| `doi` | 建议 | 有则填 |
| `arxiv_id` | 建议 | 有则填 |
| `pdf_path` | 建议 | 本地 PDF 相对路径；尚无全文可空 |
| `access_status` | 是 | 取值建议：`fulltext` / `abstract_only` / `waiting_user_pdf` / `unknown` |
| `abstract` | 建议 | 摘要（中英不限；有英文可另加 `abstract_en`，非强制） |
| `reading_notes` | 深读或入库后必填 | 对象，见下 |

### `reading_notes` 最小内容

| 子字段 | 含义 |
|--------|------|
| `problem` | 研究问题（短句） |
| `method` | 方法（短句） |
| `results` | 主要结果或结论（短句） |
| `limitations` | 局限（短句） |

可选扩展（不强制）：`data_metrics`、`quotable_claims`、证据位置等。

### 关系标注（F2 用，可选）

若记录论文间关系，每条须含：`target_id`、`kind`（如引用 / 方法继承 / 观点冲突等）、`basis`（`fact` 或 `inferred`）。推断不得写成事实。

### `cite_key` 策略（方案 A）

- **不作为入库必填字段**，也不要求本科生手填。  
- 需要写 Related Work、导出文献列表或生成 BibTeX 时，由程序按「第一作者姓（或拼音）+ 年份」等规则**自动生成**短引用键；重名冲突时自动加后缀。  
- 用户侧无感；开发者可在导出结果或 `_work/` 中看到生成值，但不得把「未填 cite_key」当成入库失败原因。

## 7. `SKILL.md` 必含章节（已确认）

每个 `SKILL.md` 按下列结构书写（可增不可缺）：

```markdown
---
name: <技能名>
description: >-
  <第三人称：做什么、何时用、关键触发词>
---

# <中文名>

## 何时使用 / 何时不使用
## 前置条件（依赖哪些技能、哪些文件须已存在）
## 硬规则（禁止项）
## 步骤（有序，可勾选）
## 必须问人的点（停下条件与如何继续）
## 交付契约（输入 / 输出路径与字段）
## 脚本调用（若有，给命令模板）
## 失败与回流
## 参考（链接 references/）
```

`description` 写清楚「做什么 + 何时用」，便于 Agent 自动发现；不要写成「我可以帮助你…」。

## 8. 技能调用关系（初版，已确认）

```text
调度模块
  → 按用户意图进入 F1 / F2 / F3 / F5
     （F4 本周未单派负责人；需要时可由写作侧降级处理或下周补）

F2 文献全景（典型顺序）
  检索下载 → 精读入库 → 全景撰写 →（可选）引用核对 → 可视化

F3 单篇深读
  → 可调用精读入库能力；结果必须回写同一 references.json

F1 入门
  → 可选调用检索；或本周降级为「内容包 / 用户给定链接」

F5 写作
  → 读取 references.json 与/或本地阶段材料
  → 成稿前必须先有 _work/claims-evidence.md

学术诚信规程
  → 全程适用，不单独生成科研正文
```

本周允许 F1/F3 在说明书中写明「调用 F2 侧已有技能」，或暂时重复步骤说明并标注后续再抽共享；重组时用 `git mv` 即可。

## 9. 工作深浅（对用户透明，方案 X）

**不对用户提供「请选择：轻量 / 标准 / 深入」之类菜单。**  
由调度根据用户话术与场景自行收敛；需要续跑时可把推断结果静默写入 `scope.md`（供 Agent 阅读），不必向用户解释档位名称。

开发者内部对照（写技能步骤时遵守）：

| 内部档 | 如何从话术推断（示例） | 行为约束 |
|--------|------------------------|----------|
| 轻量 | 「今晚组会」「快点」「先扫一眼」 | 少检索、短交付；必须标明局限；精读候选可自动但标风险 |
| 标准 | 未特别说明（默认） | 完整「必须问人」关卡；主张挂证据 |
| 深入 | 「开题」「正式相关工作」「尽量全面」 | 更广检索；关系图必出；可启用挑刺 |

未识别到轻量或深入信号时，按**标准**执行。

## 10. 完成定义 DoD（已确认）

功能分支合并进 `main` 前，负责人自检：

1. 有一句演示话术（用户说什么 → 应出现哪些文件）  
2. 产出路径与本文 §3 文件名一致  
3. 具备符合 §7 结构的 `SKILL.md`  
4. 不编造证据；推断已标明  
5. 中文交付为 UTF-8  
6. 未提交 PDF 全文、密钥、用户课题大目录  
7. 能在样例材料上当场跑通，或按步骤演示到约定交付
